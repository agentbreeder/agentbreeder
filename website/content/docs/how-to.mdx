Requires Python 3.11+:

```bash
pip3 install agentbreeder          # full CLI + API server + engine
pip3 install agentbreeder-sdk      # lightweight SDK only
```

## Scaffold a new agent

Use `agentbreeder init` to create a new agent project interactively, or use the `/agent-build` Claude Code skill for a guided advisory experience.

It supports two paths:

- **Fast Path** — you know your stack. Six quick questions, then scaffold.
- **Advisory Path** — you describe your use case. It recommends the best framework, model, RAG, memory, MCP/A2A, deployment, and eval setup — with reasoning — before scaffolding begins.

## Deploy to Different Targets

### Local (Docker Compose)

```bash
agentbreeder deploy agent.yaml --target local
```

No cloud credentials needed. Starts a Docker Compose stack on your machine.

### GCP Cloud Run

```bash
gcloud auth login
gcloud config set project my-project
agentbreeder deploy agent.yaml --target cloud-run --region us-central1
```

```yaml
deploy:
  cloud: gcp
  region: us-central1
  scaling:
    min: 0        # scale to zero when idle
    max: 10
  secrets:
    - OPENAI_API_KEY    # must exist in GCP Secret Manager
```

#### One-shot script (verified end-to-end)

The reference example `microlearning-ebook-agent` ships a `scripts/deploy_gcp.sh` that automates the full chain — APIs, Artifact Registry, Secret Manager, Cloud Build, Cloud Run — in a single command. The script bundles the agentbreeder engine alongside the agent project via `Dockerfile.cloudrun`.

```bash
cd microlearning-ebook-agent/
./scripts/deploy_gcp.sh
```

What the script does:

1. `gcloud services enable run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com secretmanager.googleapis.com`
2. Ensures the Artifact Registry repo (default `hippokit`) exists in the configured region
3. Pushes secrets (`GOOGLE_API_KEY`, `TAVILY_API_KEY`, `AGENT_AUTH_TOKEN`) to Secret Manager
4. `gcloud builds submit . --config <generated cloudbuild.yaml>` (~3 min build)
5. `gcloud run deploy <name> --image <image> --update-secrets ...`

#### IAM grants (one-time per project)

| Service account | Role | Why |
|---|---|---|
| Cloud Build SA (`<project-num>-compute@developer.gserviceaccount.com`) | `roles/storage.objectViewer` | Read source tarball |
| Cloud Build SA | `roles/cloudbuild.builds.builder` | Create builds |
| Cloud Build SA | `roles/logging.logWriter` | Write build logs |
| Cloud Run runtime SA (same default SA) | `roles/secretmanager.secretAccessor` | Mount secrets as env vars |

#### Cloud Run defaults

Region `us-central1`, 2 GiB memory, 2 vCPU, scaling `min=0` / `max=5`, concurrency 10, port 8080, `--allow-unauthenticated` (auth happens at the agent layer via bearer token — see [chat with a deployed agent](#chat-with-a-deployed-agent) below).

<Callout type="info" title="Shrink your upload">
  Add a `.gcloudignore` at the repo root to drop venvs, build artifacts, and worktrees from the source upload. On the reference project this took the upload from **578 MB to 22 MB**.
</Callout>

## Push prompts and tools to the registry

Agents reference **prompts** and **tools** from the registry instead of inlining them. Each entity has a kebab-case name, is versioned, and is resolved at agent startup (file-first — local override always wins).

### 1. Get a JWT and export it

Every management API route under `/api/v1/*` is auth-gated (247/247 routes). Bootstrap a token once per session.

```bash
# Login (or POST /api/v1/auth/register first if you don't have an account)
curl -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"you@company.com","password":"…"}'
# → {"access_token": "eyJ…", "token_type": "bearer"}

export AGENTBREEDER_API_TOKEN=eyJ…
```

The CLI reads `AGENTBREEDER_API_TOKEN` automatically.

### 2. Push a prompt

```bash
agentbreeder registry prompt push prompts/microlearning-system.md \
  --version 1.0.0 \
  --team eng \
  --description "System prompt for the microlearning ebook agent"

agentbreeder registry prompt list --team eng
```

Versions auto-bump on save — calling `update_content` on an existing prompt bumps the semver patch (e.g. `1.0.0 → 1.0.1`).

### 3. Push a tool — Python or TypeScript

The CLI auto-detects the file extension and sets the right endpoint prefix on the registry record. You don't have to specify it.

<Tabs items={['Python', 'TypeScript', 'Module path']}>
  <Tab value="Python">
```bash
agentbreeder registry tool push tools/my_tool.py \
  --description "Look up orders by ID"
# Endpoint stored as:  python:/abs/path/to/tools/my_tool.py
```
  </Tab>
  <Tab value="TypeScript">
```bash
agentbreeder registry tool push tools/my_tool.ts \
  --description "Order lookup (Node)"
# Endpoint stored as:  node:/abs/path/to/tools/my_tool.ts
```
  </Tab>
  <Tab value="Module path">
```bash
agentbreeder registry tool push engine.tools.standard.web_search
# Endpoint stored as-is for in-process Python import
```
  </Tab>
</Tabs>

```bash
agentbreeder registry tool list
```

### 4. Verify in the dashboard

| Page | What you'll see |
|---|---|
| `/prompts` | All registered prompts, with versions and a **Test** tab that runs a real Gemini call |
| `/prompts/:id` | Versioned content, render history |
| `/tools` | All registered tools, with endpoint type badges |
| `/tools/:id` | Tool detail with the **Try it** tab — see [next recipe](#run-a-registered-tool-from-cli-or-dashboard) |

## Run a registered tool from CLI or dashboard

Once a tool is in the registry, you can execute it with structured args from anywhere — CLI, dashboard, or directly via the API.

### CLI

```bash
agentbreeder registry tool run web-search \
  --args '{"query":"What is RAG?","max_results":2}'
```

### Dashboard

Navigate to `/tools/<id>` → **Try it** tab. The form is auto-generated from the tool's `SCHEMA`. Fill in arguments → click **Run** → the result panel shows `output`, `stdout`, `stderr`, `exit_code`, `duration_ms`, and any `error`.

### API (under the hood)

```
POST /api/v1/registry/tools/{tool_id}/execute
```

Auth: `deployer` role required. Returns `{output, stdout, stderr, exit_code, duration_ms, error}`.

### Endpoint dispatch — four flavors

`engine/tool_runner.py` dispatches the call based on the `endpoint` field on the tool record:

| Prefix | Dispatcher | Example tool | Typical latency |
|---|---|---|---|
| `engine.tools.standard.<name>` | In-process Python import | `markdown-writer` | ~25 ms |
| `python:<abs_path>` | Python subprocess | `get-current-time` | ~50 ms |
| `node:<abs_path>` | Node subprocess via `npx tsx` | `get-utc-time` | ~1.5 s cold |
| `http(s)://...` | HTTP POST with JSON body | (configurable) | network-bound |

The standard library ships with `web_search` (Tavily) and `markdown_writer` out of the box.

## Chat with a deployed agent

Once your agent is deployed, you can invoke it three ways: CLI, dashboard, or direct curl. Whichever path you use, runtime auth is enforced via a bearer token (`AGENT_AUTH_TOKEN`) — `/health` stays open for probes, but `/invoke` and `/stream` are gated.

### Server-side proxy (preferred)

```
POST /api/v1/agents/{id}/invoke
```

This is a thin proxy that POSTs to the agent's deployed runtime `/invoke` endpoint, attaching the bearer token from the request body or the env var `AGENT_<UPPER_SNAKE>_TOKEN`. It solves CORS and keeps secrets server-side.

Body:
```json
{
  "input": "What is your job?",
  "endpoint_url": "https://microlearning-ebook-agent-sizukgalta-uc.a.run.app",
  "auth_token": "mle-local-dev-3f7b8c2e",
  "session_id": "optional-session-id"
}
```

Returns `{output, session_id, duration_ms, status_code, error}`.

### CLI

```bash
agentbreeder registry agent invoke microlearning-ebook-agent \
  --input "What is your job?" \
  --endpoint https://microlearning-ebook-agent-sizukgalta-uc.a.run.app \
  --token $AGENT_AUTH_TOKEN
```

### Dashboard

Navigate to `/agents/:id` → **Invoke** tab. Fill in the endpoint, bearer token, and message → response panel renders the output and auto-fills `session_id` for the next turn (so multi-turn conversations stay coherent).

The `/agents` page also has a **Register from YAML** button — paste an `agent.yaml`, validate, register.

### Direct curl against the runtime

Verified live against `https://microlearning-ebook-agent-sizukgalta-uc.a.run.app`:

```bash
URL=https://microlearning-ebook-agent-sizukgalta-uc.a.run.app
TOKEN=mle-local-dev-3f7b8c2e

curl $URL/health
# {"status":"healthy","agent_name":"microlearning-ebook-agent","version":"0.1.0"}

curl -X POST $URL/invoke \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"input":"What is your job?"}'
# {"output":"My job is to turn any topic the user supplies into a polished microlearning ebook a learner can complete in 20-40 minutes.","session_id":"…","metadata":null}
```

## Stream Agent Responses

All deployed agents expose a `/stream` endpoint returning [Server-Sent Events](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events).

### curl

```bash
curl -N -X POST https://<agent-endpoint>/stream \
  -H "Content-Type: application/json" \
  -d '{"input": "Write a report on renewable energy"}'
```

### JavaScript

```javascript
const response = await fetch("https://<agent-endpoint>/stream", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ input: "Write a report on renewable energy" }),
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  for (const line of decoder.decode(value).split("\n")) {
    if (line.startsWith("data: ")) {
      const data = line.slice(6);
      if (data === "[DONE]") break;
      const event = JSON.parse(data);
      if (event.text) process.stdout.write(event.text);
    }
  }
}
```

### Python

```python
import httpx, json

with httpx.stream("POST", "https://<agent-endpoint>/stream",
                  json={"input": "Write a report"}) as r:
    for line in r.iter_lines():
        if line.startswith("data: "):
            data = line[6:]
            if data == "[DONE]": break
            event = json.loads(data)
            if "text" in event:
                print(event["text"], end="", flush=True)
```

### SSE event format by framework

| Framework | Event type | Payload |
|-----------|-----------|---------|
| Claude SDK | `data:` | `{"text": "..."}` — one event per chunk |
| CrewAI (step) | `event: step` | `{"description": "...", "result": "..."}` |
| CrewAI (final) | `event: result` | `{"output": "..."}` |
| Google ADK | `data:` | `{"text": "...", "is_final": false}` |
| All | `data:` | `[DONE]` — end of stream |

## Route Calls Through the Model Gateway

AgentBreeder ships with a self-hosted [LiteLLM proxy](/docs/gateway) that adds budget enforcement, PII guardrails, Redis caching, and live spend tracking across all your agents. Set `model.gateway: litellm` in `agent.yaml` to opt in:

```yaml
model:
  primary: claude-sonnet-4
  fallback: gpt-4o
  gateway: litellm            # route through the LiteLLM proxy at :4000
```

When the gateway is active:
- Every LLM call goes through `http://litellm:4000/v1/chat/completions` using the agent's scoped virtual key
- The LiteLLM proxy handles retries, fallbacks, rate limiting, and caching
- Cost is tracked per agent and per team — visible in the **Costs** dashboard

```bash
# Check gateway health
curl http://localhost:4000/health

# View gateway status in the dashboard
agentbreeder status --gateway
```

See [Model Gateway](/docs/gateway) for full configuration: provider routing, team budgets, Presidio PII guardrails, and OTEL traces.

## Orchestrate Multiple Agents

### Strategy overview

| Strategy | Use case |
|----------|----------|
| `router` | Classify request, route to the right agent |
| `sequential` | Agents run in order, passing state |
| `parallel` | All agents run simultaneously |
| `hierarchical` | Manager delegates to workers |
| `supervisor` | Supervisor reviews and corrects |
| `fan_out_fan_in` | Fan out to workers, aggregate results |

### Router pipeline (orchestration.yaml)

```yaml
name: support-pipeline
version: "1.0.0"
team: customer-success
strategy: router

agents:
  triage:
    ref: agents/triage-agent
    routes:
      - condition: billing   target: billing
      - condition: technical  target: technical
      - condition: default    target: general
  billing:   ref: agents/billing-agent
  technical: ref: agents/technical-agent
  general:   ref: agents/general-agent

shared_state:
  type: session_context
  backend: redis

deploy:
  target: local
```

```bash
agentbreeder orchestration validate orchestration.yaml
agentbreeder orchestration deploy orchestration.yaml
agentbreeder orchestration chat support-pipeline
```

### Programmatic (Full Code SDK)

```python
from agenthub import Orchestration

pipeline = (
    Orchestration("support-pipeline", strategy="router", team="eng")
    .add_agent("triage",  ref="agents/triage-agent")
    .add_agent("billing", ref="agents/billing-agent")
    .add_agent("general", ref="agents/general-agent")
    .with_route("triage", condition="billing", target="billing")
    .with_route("triage", condition="default", target="general")
    .with_shared_state(state_type="session_context", backend="redis")
)
pipeline.deploy()
```

## Eject from YAML to Full Code

```bash
agentbreeder eject my-agent --to code    # YAML → Python/TypeScript SDK
agentbreeder eject my-agent --to yaml    # No Code dashboard export → YAML
```

Your original `agent.yaml` is preserved. Tier mobility: No Code → Low Code → Full Code — no lock-in.

## Manage Teams and RBAC

```yaml
team: customer-success
owner: alice@company.com

access:
  visibility: team              # public | team | private
  allowed_callers:
    - team:engineering
    - team:customer-success
  require_approval: false       # true = deploys need admin approval
```

At deploy time: RBAC is checked → cost attributed to team → audit entry written. There is no way to bypass this.

## Use AgentBreeder in CI/CD

### GitHub Actions

```yaml
# .github/workflows/deploy-agent.yml
name: Deploy Agent
on:
  push:
    paths: ['agents/support-agent/**']

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate
        run: docker run --rm -v $PWD:/work -w /work rajits/agentbreeder-cli validate agents/support-agent/agent.yaml
      - name: Deploy
        run: docker run --rm -v $PWD:/work -w /work -e GOOGLE_APPLICATION_CREDENTIALS=/work/sa.json rajits/agentbreeder-cli deploy agents/support-agent/agent.yaml --target cloud-run
```

### GitLab CI

```yaml
deploy-agent:
  image: rajits/agentbreeder-cli:latest
  script:
    - agentbreeder validate agents/support-agent/agent.yaml
    - agentbreeder deploy agents/support-agent/agent.yaml --target cloud-run
```

## Run the Platform with Docker Compose

The quickest way to start the dashboard and API together is:

```bash
agentbreeder ui
```

This pulls and starts the pre-built images (dashboard, API, PostgreSQL, Redis) and opens **http://localhost:3001** automatically. Deployed agents appear in the **Agents** tab once they are registered.

Alternatively, you can start the stack directly with Docker Compose:

```bash
# Standalone — pre-built images, no repo clone needed
curl -O https://raw.githubusercontent.com/agentbreeder/agentbreeder/main/deploy/docker-compose.standalone.yml
docker compose -f docker-compose.standalone.yml up -d

# From source
docker compose -f deploy/docker-compose.yml up -d
```

| Service | URL |
|---------|-----|
| Dashboard | http://localhost:3001 |
| API | http://localhost:8000 |
| API Docs | http://localhost:8000/docs |

Default login: `admin@agentbreeder.local` / `plant`

<Callout type="warn" title="Warning">
  Change the default password before exposing this to a network.
</Callout>

### Docker networking: localhost vs host.docker.internal

From your **host terminal**, the API is at `http://localhost:8000` and the dashboard at `http://localhost:3001`.

From **inside a Docker container** (e.g. a deployed agent), `localhost` resolves to the container itself — not your host machine. Use these addresses instead:

| Host OS | Address to reach the AgentBreeder API |
|---------|--------------------------------------|
| macOS / Windows (Docker Desktop) | `http://host.docker.internal:8000` |
| Linux | `http://172.17.0.1:8000` (default Docker bridge) |

If you hard-code `localhost:8000` inside an agent container, the agent will fail to reach the API. Reference `host.docker.internal:8000` in your `agent.yaml` `env_vars` or secrets instead.

## Troubleshooting

### "agentbreeder: command not found"

This almost always means pip's script directory is not in your `PATH`.

```bash
pip3 show agentbreeder           # confirm it's installed and see Location

# macOS — find where pip installed the script:
python3 -c "import sysconfig; print(sysconfig.get_path('scripts'))"

# Add that directory to your shell (e.g. ~/.zshrc or ~/.bashrc):
export PATH="$HOME/Library/Python/3.x/bin:$PATH"   # adjust 3.x to your version
source ~/.zshrc

# Alternatively, install into a place already on PATH:
pip3 install --user agentbreeder
```

After updating PATH, open a new terminal and run `agentbreeder --help`.

### "Validation failed: unknown framework"

Supported values: `langgraph`, `openai_agents`, `claude_sdk`, `crewai`, `google_adk`, `custom`

### "RBAC check failed"

The deploying user must belong to the team in `agent.yaml`. Check `team:` matches your team membership.

### "Container build failed"

```bash
docker info                                              # check Docker is running
agentbreeder deploy agent.yaml --target local --dry-run  # inspect generated Dockerfile
```

### "Deploy rolled back"

The 8-step pipeline is atomic — any failure rolls back everything. Check which step failed:

```bash
agentbreeder status my-agent
agentbreeder logs my-agent
```

### Dashboard won't start

The dashboard needs the API running first. Start it via `docker compose up` from your local dev stack (see [Quickstart](quickstart)).
